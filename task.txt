  ---
  Как работает текущий механизм

  Kafka Consumer (TransactionIncomingWebhookAction) при каждой входящей крипто-транзакции:
  1. Ищет адрес в таблице wallet_histories — если не найден, игнорирует транзакцию
  2. Если найден — создаёт транзакцию и зачисляет баланс на user_id владельца адреса
  3. Стреляет событием AfterTransactionIncomingWebhookEvent (уже есть 4 listener-а)

  ---
  Архитектура для merchant платежей

  [Создание платежа]
    Merchant Platform
    → Находим свободный адрес в merchant_payment_addresses (пул)
    → Если нет — вызываем Crypto Gateway → новый адрес
    → Регистрируем адрес в wallet_histories (merchant user_id) ← бекенд видит его
    → Генерируем уникальный decimal suffix (0.0078) → exact_amount = 1.0078
    → Сохраняем в merchant_payments: address, exact_amount, rate, expires_at

  [Покупатель платит]
    → Отправляет 1.0078 USDT TRC-20 на адрес

  [Kafka обнаруживает транзакцию]

  [Новый listener в uexapp-backend]  ← нужно добавить
    MerchantPaymentNotificationListener
    → Проверяет: user_id транзакции принадлежит мерчанту?
    → Если да + статус SUCCESS → POST /internal/payment/incoming на merchant platform
      { address, amount, tx_hash, currency, status }

  [Merchant Platform /internal/payment/incoming]
    → Ищет merchant_payment по (address + exact_amount)
    → Обновляет статус → Success
    → Обновляет merchant_balance
    → Освобождает адрес в пуле (status → free)
    → Стреляет webhook мерчанту (DispatchMerchantWebhookJob)

  ---
  Что нужно реализовать

  В merchant platform (наше):
  - Таблица merchant_payment_addresses — пул адресов
  - Сервис вызова Crypto Gateway
  - Логика unique decimal suffix
  - /internal/payment/incoming эндпоинт
  - Job для истёкших платежей (освобождение адресов)

  В uexapp-backend (нужно добавить):
  - MerchantPaymentNotificationListener на событие AfterTransactionIncomingWebhookEvent
  - При регистрации нового адреса от merchant platform — запись в wallet_histories

  ---
  Система зачисления криптовалюты

  Общая схема

  Blockchain Node → Kafka Topic "transactions"
         ↓
  TransactionIncomingCommand (artisan consumer)
         ↓

         ↓

         ↓
  AfterTransactionIncomingWebhookEvent → email + бонусы

  ---
  Основной обработчик депозита

  src/app/Application/Actions/Webhook/TransactionWebhook/TransactionIncomingWebhookAction.php

  Последовательность выполнения:

  1. Валидация валюты — GetCurrencyByCriteriaHandler ищет по цепи и коду токена
  2. Поиск кошелька — FindWalletHandler с lockForUpdate() (защита от race condition)
  3. Дедупликация — SHA-256 хеш из chain+to+tx+amount+token, ищет существующую транзакцию
  4. Минимальный лимит — проверяет fee_limits для PaymentMethodEnum::CRYPTO
  5. Создание транзакции — CreateTransactionHandler → статус PENDING
  6. Зачисление — только если confirmations >= max_confirms AND status=true:
    - AddBalanceHandler → BalanceRepository::addBalance()
    - UpdateTransactionHandler → статус SUCCESS
  7. Событие — event(AfterTransactionIncomingWebhookEvent)

  ---
  Wallet vs WalletHistory

  Wallet — текущий баланс пользователя

  Таблица: wallets

  ┌──────────────────┬──────────────────────────────────────────────────────────┐
  │       Поле       │                        Назначение                        │
  ├──────────────────┼──────────────────────────────────────────────────────────┤
  │ user_id          │ Пользователь                                             │
  ├──────────────────┼──────────────────────────────────────────────────────────┤
  │ currency_id      │ Конкретная валюта (USDT-ERC20, BTC, ETH...)              │
  ├──────────────────┼──────────────────────────────────────────────────────────┤
  │ balance          │ Текущий баланс (строка для BC Math)                      │
  ├──────────────────┼──────────────────────────────────────────────────────────┤
  │ address          │ Текущий активный адрес для этой валюты (может быть null) │
  ├──────────────────┼──────────────────────────────────────────────────────────┤
  │ sort, is_default │ Отображение в UI                                         │
  └──────────────────┴──────────────────────────────────────────────────────────┘

  Это финансовая запись — один кошелёк на связку user_id + currency_id. Именно в нём AddBalanceHandler делает increment('balance', amount).

  ---
  WalletHistory — история адресов по цепям

  Таблица: wallet_histories

  ┌────────────────┬─────────────────────────────────────────────────────────────┐
  │      Поле      │                         Назначение                          │
  ├────────────────┼─────────────────────────────────────────────────────────────┤
  │ user_id        │ Пользователь                                                │
  ├────────────────┼─────────────────────────────────────────────────────────────┤
  │ chain_id       │ Цепь (Ethereum, Polygon, Bitcoin...) — не конкретная валюта │
  ├────────────────┼─────────────────────────────────────────────────────────────┤
  │ address        │ Адрес в этой цепи                                           │
  ├────────────────┼─────────────────────────────────────────────────────────────┤
  │ deactivated_at │ Если заполнен — адрес деактивирован                         │
  └────────────────┴─────────────────────────────────────────────────────────────┘

  Это реестр всех адресов пользователя по блокчейн-сетям. Один адрес в цепи покрывает все токены этой цепи (например, один ETH-адрес работает для USDT, USDC, DAI и самого ETH).

  ---
  Как они связаны

  POST /api/wallets/get (запрос адреса для депозита)
    ↓
  WalletRepository::create()
    ├─ WalletHistory::createOrFirst(user_id, chain_id, address)  ← сохраняет адрес
    └─ Wallet::update(['address' => $address])                   ← дублирует в Wallet для быстрого доступа

  При входящей транзакции (Kafka):
  TransactionIncomingWebhookAction
    ↓
  FindWalletHandler
    →  WalletHistory WHERE address=? AND chain_id=? AND deactivated_at IS NULL
    ←  возвращает user_id владельца адреса
    ↓
  AddBalanceHandler
    → Wallet WHERE user_id=? AND currency_id=? → increment balance

  ---
  Ключевое различие

  ┌────────────────────┬────────────────────────────────┬───────────────────────────────┐
  │                    │             Wallet             │         WalletHistory         │
  ├────────────────────┼────────────────────────────────┼───────────────────────────────┤
  │ Гранулярность      │ По валюте (currency_id)        │ По цепи (chain_id)            │
  ├────────────────────┼────────────────────────────────┼───────────────────────────────┤
  │ Главное поле       │ balance                        │ address                       │
  ├────────────────────┼────────────────────────────────┼───────────────────────────────┤
  │ Назначение         │ Хранит деньги                  │ Хранит адреса (в т.ч. старые) │
  ├────────────────────┼────────────────────────────────┼───────────────────────────────┤
  │ Поиск при депозите │ Нет — ищем по WalletHistory    │ Да — ищем владельца адреса    │
  ├────────────────────┼────────────────────────────────┼───────────────────────────────┤
  │ Обновление         │ При каждом зачислении/списании │ При выпуске нового адреса     │
  └────────────────────┴────────────────────────────────┴───────────────────────────────┘

  Зачем разделили (миграция 2025_11_20): один адрес ETH нужен сразу нескольким валютам (USDT/USDC/ETH). Хранить адрес в wallets значило бы дублировать его N раз. После разделения:
  адрес живёт в wallet_histories один раз на цепь, а wallets.address — просто кэш для UI.
